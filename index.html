GridWatch
Energy Affordability Intelligence · XGBoost + SHAP + RAG + LLM · Dr. Lanyu Shang's GenAI Course · LMU
The Problem

Walking around campus, I kept hearing the same thing — electricity bills were becoming a real burden. After researching, I found millions of households across the US struggle to pay energy bills every month, yet many don't know what assistance programs exist or how to access them. GridWatch was built to fix that — combining classical ML, explainable AI, and modern GenAI into a single end-to-end system.

Landing Page — Two User Modes
GridWatch serves two audiences — residents checking eligibility and city officials/researchers exploring risk data
Resident Flow — Eligibility Check
Step 1 of 2 — city, household size, income range (privacy-preserving ranges, no data stored)
Results show matched programs (LIHEAP, CARE, FERA, ESA) with savings estimates and direct application links
Research Dashboard — California Energy Risk Map
50 California cities mapped by ML-predicted energy risk — 18 High Risk (red) · 34 Low Risk (green) · XGBoost trained on Census, EIA, and NOAA data
City Risk Dashboard — SHAP-Powered Explainability
Per-city snapshot: energy burden %, avg household income, avg energy cost, ML risk prediction, and min temperature forecast
AI-Generated Policy Recommendations
RAG pipeline retrieves relevant government policy documents (LIHEAP, CARE, FERA, ESA, WAP) and generates city-specific, actionable recommendations — not generic advice
Conversational Research Assistant
LLM-powered assistant answers policy and data questions, citing exact SHAP values and government documents it retrieved from the vector database
What Was Built
ML Pipeline: XGBoost trained on Census Bureau, EIA, and NOAA data to predict energy burden risk across 50 California cities. SHAP explainability makes model reasoning fully transparent — showing exactly which factors (income, energy cost, temperature) drive each prediction.
RAG Pipeline: FAISS vector database embeds real government policy documents. Retrieves the most relevant documents per query, grounding every LLM response in actual program details — no hallucinations.
Research Assistant: Conversational LLM interface that answers questions about assistance programs, cites source documents, and explains risk drivers using SHAP values in plain language.

GemVault — Secure Banking App
Flask · MySQL · bcrypt · pyotp MFA · SQL Injection Protection · RBAC
Overview

GemVault is a full-stack secure banking web application built with Flask and MySQL, designed to demonstrate real-world web security practices. Every feature — from registration to money transfer — is hardened with industry-standard security measures including bcrypt password hashing, email-based MFA, SQL injection protection, input validation, and role-based access control.

Login — Secure Portal
GemVault login portal — credentials are validated against bcrypt-hashed passwords stored in MySQL
Registration — Account Creation
Users can register with Savings, Checking, or both account types — passwords are never stored in plaintext
Multi-Factor Authentication (MFA)
OTP verification page — 6-digit code sent to registered email on every login
Real OTP emails delivered via pyotp — time-based codes with expiry
How it works: After password validation, a 6-digit TOTP is generated using pyotp and emailed to the user. Login only completes after the user enters the correct OTP — protecting against stolen credentials.
Dashboard
Post-login dashboard showing account summary — access is role-gated, users can only see their own accounts
Secure Money Transfer
Transfer funds page — all inputs are validated and parameterized to prevent SQL injection attacks
Transfer Proof — Balance Before & After
Before transfer — Balance: $350
After transfer — Balance: $300 (–$50 transferred)
Security Features Implemented
bcrypt Password Hashing: Passwords are salted and hashed before storage — plaintext passwords are never saved in the database.
Email MFA via pyotp: Time-based one-time passwords sent to the user's registered email on every login attempt.
SQL Injection Protection: All database queries use parameterized statements — no raw string interpolation.
RBAC + Session Handling: Role-based access control ensures users can only access their own accounts and perform authorized actions.

Fake Review Detector
ML-Powered · Naive Bayes + TF-IDF · 40,000 Reviews · Live on Hugging Face
Overview

A machine learning web app that classifies e-commerce reviews as FAKE (computer-generated) or REAL (original). Trained on 40,000 reviews using a Multinomial Naive Bayes classifier with TF-IDF vectorization (5,000 features). The app walks through the full NLP pipeline — input → clean → TF-IDF → classify → result — and exposes linguistic signals that indicate fake reviews.

Live App — Input
Users can paste any review text or use preset examples — Suspicious 5-star, Genuine complaint, Vague generic, Detailed honest
Live App — Verdict
The model returns a verdict (REAL/FAKE) with confidence score and class probabilities after analyzing the review
Linguistic Signal Analysis
Full breakdown — text statistics, linguistic signals (excessive punctuation, all caps, vague language, extreme sentiment), and pattern matches like "amazing", "best ever", "five stars"
NLP Pipeline
Step 1 — Clean: Lowercasing, punctuation removal, stopword filtering, tokenization.
Step 2 — TF-IDF: Converts cleaned text into a 5,000-feature sparse matrix, weighting terms by frequency and inverse document frequency.
Step 3 — NB Classify: Multinomial Naive Bayes computes class probabilities and returns a FAKE or REAL verdict with confidence.
Step 4 — Linguistic Signals: Rule-based heuristics flag excessive punctuation, all-caps words, extreme sentiment, vague language, and suspicious pattern matches as additional signals.

atlas.db
PostgreSQL vs MongoDB · 622K Records · CMSI 620 — Database Systems · LMU
The Story

Built as a class project for CMSI 620 – Database Systems with Prof. Basil Latif, this was a team effort by atlas.db (Artak, Milla, Tati, and Jay). After a week of debating datasets, we landed on something we all live with every day — LA traffic collisions. We pulled ~622K records from the LA Open Data Portal and built two parallel database systems from raw CSVs for a fair head-to-head comparison.

The Dataset
LA Traffic Collision records (2010–present) · ~622,000 rows · 20 columns · sourced from LA Open Data Portal
ACID vs BASE — The Core Comparison
PostgreSQL follows the ACID model (data integrity & consistency) while MongoDB follows BASE (high availability & speed)
Data Ingestion Workflow
Both systems ingested the same raw CSV — PostgreSQL via normalization into 4 linked tables, MongoDB via denormalization into embedded collision documents
Benchmark Highlights
Geospatial Query (Q3): MongoDB won this one — counting accidents within a 1 km radius of Beverly Hills using a 2dsphere index (0.93s). PostgreSQL used PostGIS ST_Point geography but was slower on this specific query type.
Analytical Queries (Q1, Q2, Q4, Q5): PostgreSQL dominated — especially JOIN-heavy queries for complex BI analytics. Query 1 ran in just 1.72s on PostgreSQL vs 92.64s on MongoDB, demonstrating the cost of MongoDB's denormalized document lookups at scale.
The Verdict
Final score: PostgreSQL 4 — MongoDB 1 · Won 🏆 Most Effective Presentation Delivery & 🏆 Most Cohesive System Architecture
Key Takeaway

Database decisions don't stay in the backend — they surface immediately in analytics, dashboard refresh times, and decision-making. PostgreSQL's ACID guarantees made it reliable for complex BI workloads, while MongoDB's flexible schema excels for geospatial and document-centric use cases. The right database depends entirely on your workload.

Pokémon Generator
VAE vs DCGAN · Generative AI · PyTorch · 300 Epochs
Overview

This project explores two fundamental generative modeling paradigms — Variational Autoencoders (VAE) and Deep Convolutional GANs (DCGAN) — by training both on a dataset of ~2,500 Pokémon images at 64×64 resolution. The goal was to understand the core trade-off: VAE offers stable, smooth training but blurry outputs, while DCGAN produces sharper, more realistic images but is harder to train.

Architecture & Challenges
Mode Collapse Problem: Early GAN training suffered from mode collapse where the generator produced repetitive outputs. This was resolved using spectral normalization on discriminator weights, label smoothing, Two-Time Update Rule (TTUR) with different learning rates for generator and discriminator, and bilinear upsampling for stable resolution scaling. Training ran for 300 epochs on Apple Silicon MPS backend.

Key Findings
VAE — Stability
Smooth, predictable training. KL divergence and reconstruction loss converged cleanly. Outputs are blurry but structurally meaningful. Best for latent space interpolation.
DCGAN — Realism
Sharper, more vibrant outputs with richer textures. Required careful tuning to avoid mode collapse. Better for visual quality at the cost of training stability.
